# /learn-eval - 提取、评估、然后保存

扩展 `/learn` 命令，增加质量门控、保存位置决策和知识放置感知功能，在写入任何技能文件之前进行。

## 提取什么

寻找：

1. **错误解决模式** — 根本原因 + 修复方案 + 可复用性
2. **调试技术** — 非显而易见的步骤、工具组合
3. **变通方案** — 库的怪异行为、API 限制、特定版本的修复
4. **项目特定模式** — 约定、架构决策、集成模式

## 流程

1. 回顾会话以寻找可提取的模式
2. 识别最有价值/可复用的洞察

3. **确定保存位置：**
   - 问："这个模式在其他项目中有用吗？"
   - **全局**（`~/.claude/skills/learned/`）：可在 2+ 个项目中使用的通用模式（bash 兼容性、LLM API 行为、调试技术等）
   - **项目**（当前项目的 `.claude/skills/learned/`）：项目特定知识（特定配置文件的怪异行为、项目特定的架构决策等）
   - 不确定时选择全局（从全局移到项目比反向操作更容易）

4. 使用以下格式草拟技能文件：

```markdown
---
name: pattern-name
description: "130 字符以内"
user-invocable: false
origin: auto-extracted
---

# [描述性模式名称]

**提取日期：** [日期]
**上下文：** [适用场景的简要描述]

## 问题
[解决什么问题 - 要具体]

## 解决方案
[模式/技术/变通方案 - 附代码示例]

## 何时使用
[触发条件]
```

5. **质量门控 — 检查清单 + 整体裁决**

   ### 5a. 必需检查清单（通过实际读取文件来验证）

   在评估草稿之前执行以下**所有**检查：

   - [ ] 通过关键字 grep `~/.claude/skills/` 和相关项目 `.claude/skills/` 文件以检查内容重叠
   - [ ] 检查 MEMORY.md（项目和全局）是否存在重叠
   - [ ] 考虑追加到现有技能是否足够
   - [ ] 确认这是可复用模式，而非一次性修复

   ### 5b. 整体裁决

   综合检查清单结果和草稿质量，然后选择以下**之一**：

   | 裁决 | 含义 | 下一步操作 |
   |------|------|------------|
   | **保存** | 独特、具体、范围明确 | 继续步骤 6 |
   | **改进后保存** | 有价值但需要优化 | 列出改进项 → 修改 → 重新评估（一次） |
   | **合并到 [X]** | 应追加到现有技能 | 显示目标技能和添加内容 → 步骤 6 |
   | **丢弃** | 琐碎、冗余或过于抽象 | 解释原因并停止 |

   **指导维度**（为裁决提供参考，不评分）：

   - **具体性与可操作性**：包含可直接使用的代码示例或命令
   - **范围适配**：名称、触发条件和内容保持一致，聚焦于单一模式
   - **独特性**：提供现有技能未涵盖的价值（基于检查清单结果）
   - **可复用性**：未来会话中存在实际的触发场景

6. **裁决特定的确认流程**

   - **改进后保存**：展示需要的改进 + 修改后的草稿 + 更新后的检查清单/裁决（经过一次重新评估后）；如果重新评估的裁决是**保存**，则在用户确认后保存，否则遵循新裁决
   - **保存**：展示保存路径 + 检查清单结果 + 一行裁决理由 + 完整草稿 → 用户确认后保存
   - **合并到 [X]**：展示目标路径 + 添加内容（diff 格式）+ 检查清单结果 + 裁决理由 → 用户确认后追加
   - **丢弃**：仅展示检查清单结果 + 推理（无需确认）

7. 保存/合并到确定的位置

## 步骤 5 的输出格式

```
### 检查清单
- [x] skills/ grep：无重叠（或：发现重叠 → 详情）
- [x] MEMORY.md：无重叠（或：发现重叠 → 详情）
- [x] 现有技能追加：适合新文件（或：应追加到 [X]）
- [x] 可复用性：已确认（或：一次性 → 丢弃）

### 裁决：保存 / 改进后保存 / 合并到 [X] / 丢弃

**理由：**（1-2 句话解释裁决）
```

## 设计理由

此版本取代了之前的 5 维度数字评分标准（具体性、可操作性、范围适配、非冗余性、覆盖率各评 1-5 分），采用基于检查清单的整体裁决系统。现代前沿模型（Opus 4.6+）具有强大的上下文判断能力 — 将丰富的定性信号强制塞入数字分数会丢失细微差别，可能产生误导性的总分。整体方法让模型自然地权衡所有因素，产生更准确的保存/丢弃决策，同时明确的检查清单确保不会遗漏关键检查。

## 注意事项

- 不要提取琐碎的修复（错别字、简单语法错误）
- 不要提取一次性问题（特定 API 宕机等）
- 专注于能在未来会话中节省时间的模式
- 保持技能聚焦 — 每个技能一个模式
- 当裁决为合并时，追加到现有技能而不是创建新文件

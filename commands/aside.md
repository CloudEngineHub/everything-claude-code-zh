# Aside 命令

在任务进行中提问并获取即时、聚焦的回答——然后从断点继续。当前任务、文件和上下文不会被修改。

## 何时使用

- 当 Claude 正在工作时，你想了解某些事情但又不想打断进度
- 你需要快速解释 Claude 正在编辑的代码
- 你想在不偏离任务的情况下获得决策的第二意见或澄清
- 你需要在 Claude 继续之前理解一个错误、概念或模式
- 你想问与当前任务无关的问题，而又不想开启新的会话

## 用法

```
/aside <你的问题>
/aside 这个函数实际上返回什么？
/aside 这个模式是线程安全的吗？
/aside 为什么我们在这里使用 X 而不是 Y？
/aside foo() 和 bar() 有什么区别？
/aside 我们刚添加的 N+1 查询需要担心吗？
```

## 流程

### 步骤 1：冻结当前任务状态

在回答任何问题之前，在脑海中记录：
- 当前活跃任务是什么？（正在处理什么文件、功能或问题）
- 调用 `/aside` 时正在执行什么步骤？
- 下一步计划做什么？

在 aside 期间不要触碰、编辑、创建或删除任何文件。

### 步骤 2：直接回答问题

以仍然完整且有用的最简洁形式回答问题。

- 先给答案，再给推理过程
- 保持简短——如果需要完整解释，提议在任务完成后再深入
- 如果问题涉及当前正在处理的文件或代码，精确引用（文件路径和行号，如相关）
- 如果回答需要读取文件，可以读取——但只能读取，绝不写入

将回答格式化为：

```
ASIDE: [简要重述问题]

[你的回答]

— 恢复任务: [正在做的事情的一行描述]
```

### 步骤 3：恢复主任务

提供回答后，立即从暂停的确切位置继续活跃任务。除非 aside 回答揭示了一个阻碍或有理由重新考虑当前方法的原因（参见边缘情况），否则不要请求许可再恢复。

## 边缘情况

**未提供问题（`/aside` 后面什么都没有）：**
回复：
```
ASIDE: 未提供问题

你想了解什么？（提出你的问题，我会在不丢失当前任务上下文的情况下回答）

— 恢复任务: [正在做的事情的一行描述]
```

**问题揭示当前任务的潜在问题：**
在恢复之前明确标记：
```
ASIDE: [回答]

WARNING: 注意：此回答表明当前方法存在 [问题]。想在继续之前处理这个问题，还是按计划继续？
```
等待用户的决定后再恢复。

**问题实际上是任务重定向（不是旁注问题）：**
如果问题暗示改变正在构建的内容（例如 `/aside 实际上，我们改用 Redis 吧`），请澄清：
```
ASIDE: 这听起来像是方向变更，不只是一个旁注问题。
你想：
  (a) 仅作为信息回答，保持当前计划
  (b) 暂停当前任务并改变方法
```
等待用户的回答——不要自行假设。

**问题关于当前打开的文件或代码：**
从实时上下文回答。如果文件在之前的会话中已读取，直接引用。如果没有，现在读取（只读）并附上 file:line 引用回答。

**没有活跃任务（调用 `/aside` 时没有正在进行的任务）：**
仍然使用标准包装格式以保持响应格式一致：
```
ASIDE: [简要重述问题]

[你的回答]

— 恢复任务: 没有活跃任务可恢复
```

**问题需要长回答：**
简洁地给出要点回答，然后提议：
```
这是简短版本。想在我们完成 [当前任务] 后获得更深入的解释吗？
```

**连续多个 `/aside` 问题：**
按顺序回答每个问题。最后一个回答后，恢复主任务。不要在一连串 aside 中丢失任务状态。

**Aside 回答暗示需要进行代码更改：**
记录需要的更改，但在 aside 期间不要执行：
```
ASIDE: [回答]

 值得修复: [应该更改的内容]。除非你现在想处理，否则我会在当前任务完成后标记它。
```

**问题模糊或过于笼统：**
问一个澄清问题——获取回答所需信息的最短问题。不要问多个问题。

## 示例输出

```
用户: /aside fetchWithRetry() 实际上做了什么？

ASIDE: fetchWithRetry() 做了什么？

fetchWithRetry() (src/api/retry.ts:12) 尝试最多 3 次请求，使用指数退避（250ms → 500ms → 1s）。
它只在 5xx 响应和网络错误时重试——4xx 错误被视为最终结果，不会重试。

— 恢复任务: 重构 src/middleware/auth.ts 中的认证中间件
```

```
用户: /aside 我们采用的方法是线程安全的吗？

ASIDE: 当前方法是线程安全的吗？

不是——src/cache/store.ts:34 中的共享缓存对象在没有加锁的情况下被修改。
在单进程 Node.js 服务器中，这是低风险的竞态条件，但在使用工作线程或集群时会成为真正的问题。

WARNING: 注意：这可能影响我们正在构建的功能。想现在处理还是继续并在后续修复？
```

## 注意事项

- 在 aside 期间绝不修改文件——仅限只读访问
- Aside 是对话暂停，不是新任务——原始任务必须始终恢复
- 保持回答聚焦：目标是快速解除用户的阻碍，而不是做讲座
- 如果 aside 引发了更大的讨论，先完成当前任务，除非 aside 揭示了阻碍
- Aside 不会保存到会话文件，除非明确与任务结果相关

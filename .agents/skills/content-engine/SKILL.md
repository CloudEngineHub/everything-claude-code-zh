# 内容引擎

构建平台原生内容，而不将作者的真实声音扁平化为平台废话。

## 何时激活

- 撰写 X 帖子或线程
- 起草 LinkedIn 帖子或发布更新
- 编写短视频或 YouTube 讲解脚本
- 将文章、播客、演示、文档或内部笔记重新利用为公共内容
- 围绕产品、洞察或叙述构建发布序列或持续的内容系统

## 不可协商

1. 从源材料开始，而不是通用帖子公式。
2. 适应平台的格式，而不是人格。
3. 一个帖子应该承载一个真正的声明。
4. 具体性优于形容词。
5. 除非用户明确要求，否则没有参与诱饵。

## 来源优先工作流

在起草之前，识别来源集：
- 已发表的文章
- 笔记或内部备忘录
- 产品演示
- 文档或变更日志
- 记录稿
- 屏幕截图
- 来自同一作者的先前帖子

如果用户想要特定的声音，请在写作之前从真实示例构建声音配置文件。
当跨多个输出的声音一致性重要时，使用 `brand-voice` 作为规范工作流。

## 声音处理

`brand-voice` 是规范的声音层。

在以下情况下首先运行它：

- 有多个下游输出
- 用户明确关心写作风格
- 内容是发布、外展或声誉敏感的

在此处重用产生的 `VOICE PROFILE`，而不是重建第二个声音模型。
如果用户特别想要 Affaan / ECC 声音，仍然将 `brand-voice` 视为事实来源，并向其提供可用的最佳实时或源衍生材料。

## 严格禁止

删除并重写以下任何内容：
- "在当今快速发展的格局中"
- "游戏规则改变者"、"革命性"、"尖端"
- "这就是为什么这很重要"，除非紧随其后的是具体内容
- 以 LinkedIn 风格的问题结束只是为了培育回复
- LinkedIn 上的强制随意性
- 来源材料中不存在的假参与填充

## 平台适应规则

### X

- 以最强的声明、工件或紧张开场
- 如果源声音是压缩的，则保持压缩
- 如果编写线程，每个帖子必须推进论点
- 不要用受众不需要的上下文填充

### LinkedIn

- 仅扩展到直接细分之外的人们可以跟随的程度
- 除非源材料实际上是反思的，否则不要将其变成假教训帖子
- 没有企业灵感节奏
- 没有赞美堆叠，没有"旅程"填充

### 短视频

- 围绕视觉序列和证明点编写脚本
- 前几秒应该显示结果、问题或冲击
- 不要编写在纸上听起来比屏幕上更好的旁白

### YouTube

- 早期显示结果或紧张
- 按论点或进展组织，而不是填充部分
- 仅在有助于清晰度时使用章节

### 新闻简报

- 以要点、冲突或工件开场
- 不要花第一段热身
- 每个部分都需要添加新内容

## 重新利用流程

1. 选择锚定资产。
2. 提取 3 到 7 个原子声明或场景。
3. 按锐度、新颖性和证明对它们进行排名。
4. 为每个输出分配一个强有力的想法。
5. 适应每个平台的结构。
6. 去除平台形状的填充。
7. 运行质量门。

## 交付物

当被要求提供活动时，返回：
- 如果声音匹配很重要，则提供简短的声音配置文件
- 核心角度
- 平台原生草稿
- 发布顺序（仅当有助于执行时）
- 发布前必须填补的缺口

## 质量门

在交付之前：
- 每个草稿听起来像预期作者，而不是平台刻板印象
- 每个草稿包含真正的声明、证明点或具体观察
- 没有通用炒作语言保留
- 没有假参与诱饵保留
- 除非要求，否则跨平台没有重复的副本
- 任何 CTA 都是 earned 且用户批准的

## 相关技能

- `brand-voice` 用于来源衍生的声音配置文件
- `crosspost` 用于特定平台的分发
- `x-api` 用于获取最近的帖子和发布批准的 X 输出

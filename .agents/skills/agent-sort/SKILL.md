# Agent 排序

当仓库需要项目特定的 ECC 表面而不是默认的完整安装时，请使用此技能。

目标不是猜测什么"感觉有用"。目标是使用实际代码库的证据对 ECC 组件进行分类。

## 何时使用

- 项目只需要 ECC 的子集，完整安装太嘈杂
- 仓库堆栈很清楚，但没有人想逐一手工精选技能
- 团队想要可重复的安装决策，由 grep 证据支持而非意见
- 需要将始终加载的日常工作流表面与可搜索的库/参考表面分开
- 仓库已漂移到错误的语言、规则或钩子集，需要清理

## 不可协商的规则

- 使用当前仓库作为事实来源，而不是通用偏好
- 每个 DAILY 决策必须引用具体的仓库证据
- LIBRARY 并不意味着"删除"；它意味着"保持可访问而不默认加载"
- 不要安装当前仓库无法使用的钩子、规则或脚本
- 优先使用 ECC 原生表面；不要引入第二个安装系统

## 输出

按顺序生成这些工件：

1. DAILY 清单
2. LIBRARY 清单
3. 安装计划
4. 验证报告
5. 可选的 `skill-library` 路由器（如果项目需要）

## 分类模型

仅使用两个存储桶：

- `DAILY`
  - 应在此仓库的每个会话中加载
  - 与仓库的语言、框架、工作流或操作者表面强烈匹配
- `LIBRARY`
  - 值得保留，但不足以默认加载
  - 应仍可通过搜索、路由器技能或选择性手动使用来访问

## 证据来源

在进行任何分类之前使用仓库本地证据：

- 文件扩展名
- 包管理器和锁定文件
- 框架配置
- CI 和钩子配置
- 构建/测试脚本
- 导入和依赖清单
- 明确描述堆栈的仓库文档

有用的命令包括：

```bash
rg --files
rg -n "typescript|react|next|supabase|django|spring|flutter|swift"
cat package.json
cat pyproject.toml
cat Cargo.toml
cat pubspec.yaml
cat go.mod
```

## 并行审查传递

如果有并行子代理可用，将审查拆分为这些传递：

1. Agents
   - 分类 `agents/*`
2. Skills
   - 分类 `skills/*`
3. Commands
   - 分类 `commands/*`
4. Rules
   - 分类 `rules/*`
5. 钩子和脚本
   - 分类钩子表面、MCP 健康检查、辅助脚本和操作系统兼容性
6. 额外内容
   - 分类上下文、示例、MCP 配置、模板和指南文档

如果子代理不可用，则按顺序运行相同的传递。

## 核心工作流

### 1. 读取仓库

在对任何内容进行分类之前建立真正的堆栈：

- 使用的语言
- 使用的框架
- 主要包管理器
- 测试堆栈
- lint/格式堆栈
- 部署/运行时表面
- 已存在的操作者集成

### 2. 构建证据表

对于每个候选表面，记录：

- 组件路径
- 组件类型
- 提议的存储桶
- 仓库证据
- 简短理由

使用此格式：

```text
skills/frontend-patterns | skill | DAILY | 84 个 .tsx 文件，存在 next.config.ts | 核心前端堆栈
skills/django-patterns   | skill | LIBRARY | 无 .py 文件，无 pyproject.toml       | 在此仓库中不活动
rules/typescript/*       | rules | DAILY | package.json + tsconfig.json            | 活动 TS 仓库
rules/python/*           | rules | LIBRARY | 零个 Python 源文件             | 仅保持可访问
```

### 3. 决定 DAILY 与 LIBRARY

在以下情况下提升为 `DAILY`：

- 仓库清楚使用匹配的堆栈
- 组件足够通用以帮助每个会话
- 仓库已经依赖于相应的运行时或工作流

在以下情况下降级为 `LIBRARY`：

- 组件不在堆栈上
- 仓库以后可能需要它，但不是每天
- 它增加了上下文开销而没有直接相关性

### 4. 构建安装计划

将分类转化为操作：

- DAILY 技能 -> 在 `.claude/skills/` 中安装或保留
- DAILY 命令 -> 仅在仍然有用时保留为显式填充
- DAILY 规则 -> 仅安装匹配的语言集
- DAILY 钩子/脚本 -> 仅保留兼容的
- LIBRARY 表面 -> 通过搜索或 `skill-library` 保持可访问

如果仓库已经使用选择性安装，请更新该计划而不是创建另一个系统。

### 5. 创建可选的库路由器

如果项目想要可搜索的库表面，请创建：

- `.claude/skills/skill-library/SKILL.md`

该路由器应包含：
- DAILY 与 LIBRARY 的简短说明
- 分组的触发关键字
- 库引用所在的位置

不要在路由器中重复每个技能正文。

### 6. 验证结果

应用计划后，验证：
- 每个 DAILY 文件都存在于预期位置
- 陈旧的语言规则未处于活动状态
- 不兼容的钩子未安装
- 产生的安装实际上匹配仓库堆栈

返回包含以下内容的紧凑报告：
- DAILY 计数
- LIBRARY 计数
- 删除的陈旧表面
- 未决问题

## 交接

如果下一步是交互式安装或修复，请移交给：
- `configure-ecc`

如果下一步是重叠清理或目录审查，请移交给：
- `skill-stocktake`

如果下一步是更广泛的上下文修剪，请移交给：
- `strategic-compact`

## 输出格式

按此顺序返回结果：

```text
STACK
- 语言/框架/运行时摘要

DAILY
- 始终加载的项目及其证据

LIBRARY
- 可搜索/参考项目及其证据

INSTALL PLAN
- 应安装、删除或路由的内容

VERIFICATION
- 运行的检查和剩余的缺口
```

# SEO

通过技术正确性、性能和内容相关性来提升搜索可见性，而非取巧手段。

## 何时使用

在以下情况使用此技能：
- 审计可抓取性、可索引性、规范链接或重定向
- 改进标题标签、元描述和标题结构
- 添加或验证结构化数据
- 改进 Core Web Vitals
- 进行关键词研究并将关键词映射到 URL
- 规划内链或 Sitemap / Robots 更改

## 工作原理

### 原则

1. 在内容优化之前先修复技术障碍。
2. 一个页面应该有一个清晰的主要搜索意图。
3. 优先考虑长期质量信号而非操纵性模式。
4. 移动优先假设很重要，因为索引是移动优先的。
5. 建议应该是针对具体页面且可实施的。

### 技术 SEO 清单

#### 可抓取性

- `robots.txt` 应允许重要页面并阻止低价值页面
- 没有重要页面被无意设为 `noindex`
- 重要页面应在浅层点击深度内可达
- 避免超过两次跳转的重定向链
- 规范标签应自洽且不形成循环

#### 可索引性

- 首选 URL 格式应保持一致
- 多语言页面如果使用的话需要正确的 hreflang
- Sitemap 应反映预期的公开面
- 不应有竞争的重复 URL 而没有规范控制

#### 性能

- LCP < 2.5s
- INP < 200ms
- CLS < 0.1
- 常见修复：预加载关键资源、减少渲染阻塞工作、预留布局空间、精简重型 JS

#### 结构化数据

- 首页：适用时使用 Organization 或 Business Schema
- 编辑页面：`Article` / `BlogPosting`
- 产品页面：`Product` 和 `Offer`
- 内页：`BreadcrumbList`
- 问答部分：仅在内容真正匹配时使用 `FAQPage`

### 页面规则

#### 标题标签

- 目标约 50-60 个字符
- 将主要关键词或概念放在前面
- 让标题对人类可读，而非为机器人堆砌关键词

#### 元描述

- 目标约 120-160 个字符
- 诚实描述页面内容
- 自然地包含主要主题

#### 标题结构

- 一个清晰的 `H1`
- `H2` 和 `H3` 应反映实际内容层级
- 不要仅为了视觉样式而跳过结构

### 关键词映射

1. 定义搜索意图
2. 收集现实的关键词变体
3. 按意图匹配度、可能价值和竞争度排列优先级
4. 将一个主要关键词/主题映射到一个 URL
5. 检测并避免自相残杀

### 内链

- 从强页面链接到你想要排名的页面
- 使用描述性的锚文本
- 当更具体的锚文本可用时避免通用锚文本
- 从新页面反向填充链接到相关的现有页面

## 示例

### 标题公式

```text
主要主题 - 特定修饰词 | 品牌
```

### 元描述公式

```text
行动 + 主题 + 价值主张 + 一个支撑细节
```

### JSON-LD 示例

```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "页面标题",
  "author": {
    "@type": "Person",
    "name": "作者姓名"
  },
  "publisher": {
    "@type": "Organization",
    "name": "品牌名称"
  }
}
```

### 审计输出格式

```text
[高] 产品页面上的重复标题标签
位置：src/routes/products/[slug].tsx
问题：动态标题坍缩为相同的默认字符串，削弱了相关性并产生重复信号。
修复：使用产品名称和主要类别为每个产品生成唯一标题。
```

## 反模式

| 反模式 | 修复 |
| --- | --- |
| 关键词堆砌 | 首先为用户而写 |
| 薄弱的近似重复页面 | 合并或区分它们 |
| 实际不存在的 Schema 内容 | 让 Schema 与现实匹配 |
| 不检查实际页面的内容建议 | 先阅读真实页面 |
| 泛泛的"改善 SEO"输出 | 将每个建议关联到具体页面或资产 |

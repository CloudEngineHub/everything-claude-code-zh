# 仪表板构建器

当任务是构建一个可运营的仪表板时使用此技能。

目标不是"展示每个指标"。目标是回答：

- 系统健康吗？
- 瓶颈在哪里？
- 发生了什么变化？
- 某人应该采取什么行动？

## 何时使用

- "构建一个 Kafka 监控仪表板"
- "为 Elasticsearch 创建一个 Grafana 仪表板"
- "为这个服务制作一个 SigNoz 仪表板"
- "将此指标列表转化为真正的运营仪表板"

## 防护栏

- 不要从视觉布局开始；从运营问题开始
- 不要仅因为存在就包含每个可用指标
- 不要在没有结构的情况下混合健康、吞吐量和资源面板
- 不要发布没有标题、单位和合理阈值的面板

## 工作流

### 1. 定义运营问题

围绕以下组织：

- 健康 / 可用性
- 延迟 / 性能
- 吞吐量 / 容量
- 饱和度 / 资源
- 服务特定风险

### 2. 研究目标平台模式

首先检查现有仪表板：

- JSON 结构
- 查询语言
- 变量
- 阈值样式
- 区域布局

### 3. 构建最小可用面板

推荐结构：

1. 概览
2. 性能
3. 资源
4. 服务特定区域

### 4. 削减展示面板

每个面板都应该回答一个真实问题。如果不能，删除它。

## 示例面板集

### Elasticsearch

- 集群健康
- 分片分配
- 搜索延迟
- 索引速率
- JVM 堆 / GC

### Kafka

- 代理数量
- 未复制的分区
- 消息流入 / 流出
- 消费者延迟
- 磁盘和网络压力

### API 网关 / 入口

- 请求速率
- p50 / p95 / p99 延迟
- 错误率
- 上游健康
- 活跃连接数

## 质量检查清单

- [ ] 有效的仪表板 JSON
- [ ] 清晰的区域分组
- [ ] 标题和单位已存在
- [ ] 阈值/状态颜色有意义
- [ ] 存在常用过滤器的变量
- [ ] 默认时间范围和刷新间隔合理
- [ ] 没有无运营价值的展示面板

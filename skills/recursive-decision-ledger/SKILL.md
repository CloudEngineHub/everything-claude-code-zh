# 递归决策账本

当用户试图通过重复推出或"Prime Gauss"风格的递归提示来强制更深入的计算时使用此技能。保留有用的部分：重复试验、先验记忆、新鲜信息和显式标记。移除不安全的的部分：假装循环证明确定性。

## 账本契约

每次推出应记录：

- 推出 ID 和时间戳；
- 先前接受的获胜者和先前观察列表；
- 摄入的新鲜信息；
- 搜索空间大小；
- 使用的模型族或启发式方法；
- 试验次数和有效试验次数；
- 顶级候选者；
- 决策标记；
- 针对先前账本的一致性标记；
- 提升门控结果。

首选 JSONL 用于仅追加账本，Markdown 用于人类摘要。

## 推出循环

1. 加载先前账本。
2. 在时间步零捕获新信息。
3. 运行有界搜索。
4. 标记每个候选者：接受、观察、拒绝、衰减观察或需要重放。
5. 将获胜者与先前获胜者和最新标记的推出进行比较。
6. 当漂移、尾部风险、过时数据或失败的重放使先前标记无效时，降级候选者。
7. 在汇总之前追加产物。

## 一致性标记

包含一个紧凑的一致性标记：

```text
集成匹配先前获胜者：true
递归匹配先前获胜者：false
最新推出匹配：true
允许实时提升：false
原因：重放和新鲜度门控未满足
```

## 提升规则

对于交易、资本分配、生产部署、迁移或破坏性操作，递归置信度不是批准。

默认为模拟、空运行、只读、预览或分阶段模式，除非用户明确批准实时操作且仓库/服务门控支持。

仅在以下情况下提升：

- 候选者在所选指标上击败先前接受的获胜者；
- 正确性和重放检查通过；
- 风险限制是明确的；
- 证据是持久的；
- 用户在需要时已批准实时步骤。

## 摘要格式

以决策开头，而非戏剧：

```text
推出 15 完成。先前获胜者仍然成立，但优势劣化了 17%。
状态：观察，非实时。下一个门控：20 次重放填充且新鲜订单簿年龄低于阈值。
```

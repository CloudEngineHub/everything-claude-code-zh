# 智能体排序

当存储库需要项目特定的 ECC 表面而非默认完整安装时，使用此技能。

目标不是猜测什么"感觉有用"。目标是使用来自实际代码库的证据对 ECC 组件进行分类。

## 何时使用

- 项目只需要 ECC 的子集，完整安装太嘈杂
- 存储库堆栈清晰，但没有人想手动逐一策划技能
- 团队想要由 grep 证据而非意见支持的可重复安装决策
- 需要分离始终加载的日常工作流表面与可搜索的库/参考表面
- 存储库已漂移到错误的语言、规则或挂钩集，需要清理

## 不可协商的规则

- 使用当前存储库作为真实来源，而非通用偏好
- 每个 DAILY 决策必须引用具体的存储库证据
- LIBRARY 不意味着"删除"；它意味着"保持可访问而无需默认加载"
- 不要安装当前存储库无法使用的挂钩、规则或脚本
- 优先考虑 ECC 原生表面；不要引入第二个安装系统

## 输出

按顺序生成这些工件：

1. DAILY 库存
2. LIBRARY 库存
3. 安装计划
4. 验证报告
5. 可选的 `skill-library` 路由器（如果项目需要）

## 分类模型

仅使用两个存储桶：

- `DAILY`
  - 应为此存储库的每个会话加载
  - 与存储库的语言、框架、工作流或操作员表面强烈匹配
- `LIBRARY`
  - 值得保留，但不值得默认加载
  - 应通过搜索、路由器技能或选择性手动使用保持可访问

## 证据来源

在进行任何分类之前使用存储库本地证据：

- 文件扩展名
- 包管理器和锁定文件
- 框架配置
- CI 和挂钩配置
- 构建/测试脚本
- 导入和依赖清单
- 明确描述堆栈的存储库文档

有用的命令包括：

```bash
rg --files
rg -n "typescript|react|next|supabase|django|spring|flutter|swift"
cat package.json
cat pyproject.toml
cat Cargo.toml
cat pubspec.yaml
cat go.mod
```

## 并行审查传递

如果有并行子智能体，将审查拆分为这些传递：

1. 智能体
   - 分类 `agents/*`
2. 技能
   - 分类 `skills/*`
3. 命令
   - 分类 `commands/*`
4. 规则
   - 分类 `rules/*`
5. 挂钩和脚本
   - 分类挂钩表面、MCP 健康检查、助手脚本和 OS 兼容性
6. 附加组件
   - 分类上下文、示例、MCP 配置、模板和指导文档

如果子智能体不可用，按顺序运行相同的传递。

## 核心工作流程

### 1. 读取存储库

在对任何内容进行分类之前建立真实堆栈：

- 使用的语言
- 使用的框架
- 主包管理器
- 测试堆栈
- lint/格式堆栈
- 部署/运行时表面
- 已存在的操作员集成

### 2. 构建证据表

对于每个候选表面，记录：

- 组件路径
- 组件类型
- 提议的存储桶
- 存储库证据
- 简短理由

使用此格式：

```text
skills/frontend-patterns | skill | DAILY | 84 .tsx 文件，存在 next.config.ts | 核心前端堆栈
skills/django-patterns   | skill | LIBRARY | 没有 .py 文件，没有 pyproject.toml       | 此存储库中未激活
rules/typescript/*       | rules | DAILY | package.json + tsconfig.json            | 活跃 TS 存储库
rules/python/*           | rules | LIBRARY | 零个 Python 源文件             | 仅保持可访问
```

### 3. 决定 DAILY 与 LIBRARY

当以下情况时提升到 `DAILY`：

- 存储库清楚地使用匹配的堆栈
- 组件足够通用以帮助每个会话
- 存储库已经依赖相应的运行时或工作流

当以下情况时降级到 `LIBRARY`：

- 组件脱离堆栈
- 存储库稍后可能需要它，但不是每天
- 它增加上下文开销而没有直接相关性

### 4. 构建安装计划

将分类转换为操作：

- DAILY 技能 -> 安装或保持在 `.claude/skills/`
- DAILY 命令 -> 仅在仍有用时保持为显式 shims
- DAILY 规则 -> 仅安装匹配的语言集
- DAILY 挂钩/脚本 -> 仅保留兼容的
- LIBRARY 表面 -> 通过搜索或 `skill-library` 保持可访问

如果存储库已使用选择性安装，则更新该计划而非创建另一个系统。

### 5. 创建可选的库路由器

如果项目想要可搜索的库表面，创建：

- `.claude/skills/skill-library/SKILL.md`

该路由器应包含：
- DAILY 与 LIBRARY 的简短解释
- 分组触发关键字
- 库引用所在位置

不要在路由器中重复每个技能主体。

### 6. 验证结果

应用计划后，验证：

- 每个 DAILY 文件存在于预期位置
- 过时的语言规则未保持活动
- 未安装不兼容的挂钩
- 结果安装实际上匹配存储库堆栈

返回包含以下内容的紧凑报告：

- DAILY 计数
- LIBRARY 计数
- 删除的过时表面
- 未决问题

## 移交

如果下一步是交互式安装或修复，移交给：

- `configure-ecc`

如果下一步是重叠清理或目录审查，移交给：

- `skill-stocktake`

如果下一步是更广泛的上下文修剪，移交给：

- `strategic-compact`

## 输出格式

按此顺序返回结果：

```text
STACK
- 语言/框架/运行时摘要

DAILY
- 始终加载的项目及证据

LIBRARY
- 可搜索/参考项目及证据

INSTALL PLAN
- 应安装、删除或路由的内容

VERIFICATION
- 运行的检查和剩余差距
```

# 邮件运维

当真实任务是邮箱工作时使用：分类、草拟、回复、发送或证明邮件已进入已发送箱。

这不是通用写作技能。它是围绕实际邮件接口的运维工作流。

## 技能栈

在相关时将这些 ECC 原生技能拉入工作流：

- `brand-voice` 在草拟任何面向用户的内容之前
- `investor-outreach` 用于面向投资者、合作伙伴或赞助商的邮件
- `customer-billing-ops` 当对话线程是计费/支持事件而非普通通信时
- `knowledge-ops` 当消息或对话线程应在事后捕获到持久上下文中时
- `research-ops` 当回复依赖于新鲜的外部事实时

## 何时使用

- 用户要求分类收件箱或归档低信号邮件
- 用户想要草稿、回复或新的外发邮件
- 用户想知道某封邮件是否已发送
- 用户想要证明使用了哪个账户、线程或已发送条目

## 防护措施

- 除非用户明确要求实时发送，否则先草拟
- 永远不要在没有真实已发送文件夹或客户端确认的情况下声称消息已发送
- 不要随意切换发件人账户；选择与项目和收件人匹配的账户
- 在清理过程中不要删除不确定的商业邮件
- 如果任务实际上是 DM 或 iMessage 工作，移交给 `messages-ops`

## 工作流

### 1. 确定确切的接口

在行动之前，确定：

- 哪个邮箱账户
- 哪个对话线程或收件人
- 任务是分类、草拟、回复还是发送
- 用户想要仅草拟还是实时发送

### 2. 在撰写之前阅读对话线程

如果回复：

- 阅读现有对话线程
- 确定最后一次外联
- 确定任何承诺、截止日期或未回答的问题

如果创建新外发：

- 确定热情程度
- 选择正确的渠道和发件人账户
- 在草拟之前拉取 `brand-voice`

### 3. 草拟，然后验证

对于仅草拟工作：

- 生成最终文本
- 说明发件人、收件人、主题和目的

对于实时发送工作：

- 首先验证确切的最终正文
- 通过选定的邮件接口发送
- 确认消息已进入已发送或等效的发送副本存储

### 4. 报告确切状态

使用确切的状态词：

- drafted（已草拟）
- approval-pending（待批准）
- sent（已发送）
- blocked（被阻止）
- awaiting verification（等待验证）

如果发送接口被阻止，保留草稿并报告确切的阻碍，而不是在未说明的情况下即兴使用第二种传输方式。

## 输出格式

```text
邮件接口
- 账户
- 对话线程 / 收件人
- 请求的操作

草稿
- 主题
- 正文

状态
- drafted / sent / blocked
- 适用时的已发送证明

下一步
- 发送
- 跟进
- 归档 / 移动
```

## 常见陷阱

- 不要在没有发送副本检查的情况下声称发送成功
- 不要忽略对话线程历史并写出无上下文的回复
- 不要将邮箱工作与 DM 或短信工作流混在一起
- 不要暴露密钥、认证详情或不必要的消息元数据

## 验证

- 响应命名了账户和对话线程或收件人
- 任何发送声明包含已发送证明或明确的客户端确认
- 最终状态是 drafted / sent / blocked / awaiting verification 之一

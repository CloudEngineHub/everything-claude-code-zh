# 关系优化器

重新组织用户的网络，而非将外联视为单向的潜在客户列表。

此技能处理：

- X 关注清理和扩展
- LinkedIn 关注和连接分析
- 审查优先的修剪队列
- 添加和关注推荐
- 温暖路径识别
- 以用户真实声音起草的 Apple Mail、X DM 和 LinkedIn 消息

## 何时激活

- 用户想要修剪他们的 X 关注列表
- 用户想要重新平衡他们的关注或保持连接的对象
- 用户说"清理我的网络"、"我应该取消关注谁"、"我应该关注谁"、"我应该重新联系谁"
- 外联质量取决于网络结构，而非仅是冷列表生成

## 必需输入

收集或推断：

- 当前优先级和活跃工作
- 目标角色、行业、地区或生态系统
- 平台选择：X、LinkedIn 或两者
- 不触碰列表
- 模式：`light-pass`、`default` 或 `aggressive`

如果用户未指定模式，使用 `default`。

## 工具要求

### 首选

- `x-api` 用于 X 图谱检查和最近活动
- `lead-intelligence` 用于目标发现和温暖路径排名
- `social-graph-ranker` 当用户想要独立于更广泛潜在客户工作流的桥接价值评分时
- Exa / 深度研究用于人员和公司丰富
- `brand-voice` 在起草外联之前

### 备选

- 浏览器控制用于 LinkedIn 分析和起草
- 浏览器控制用于 X（如果 API 覆盖受限）
- Apple Mail 或 Mail.app 通过桌面自动化起草（当邮件是正确渠道时）

## 安全默认值

- 默认是审查优先，绝不盲目自动修剪
- X：仅修剪用户关注的账号，绝不修剪关注者
- LinkedIn：将 1 度连接移除视为手动审查优先
- 不自动发送 DM、邀请或邮件
- 在任何应用步骤之前输出排名行动方案和草稿

## 平台规则

### X

- 互相关注意味着比单向关注更牢固
- 未回关的可以更积极地修剪
- 高度不活跃或消失的账号应快速浮出
- 互动、信号质量和桥接价值比原始粉丝数更重要

### LinkedIn

- 如果用户确实有 LinkedIn API 访问权限，优先使用 API
- 当 API 访问缺失时，浏览器工作流必须可用
- 区分出站关注和已接受的 1 度连接
- 出站关注可以更自由地修剪
- 已接受的 1 度连接应默认为审查，而非自动移除

## 模式

### `light-pass`

- 仅修剪高置信度的低价值单向关注
- 其余浮出供审查
- 生成少量添加/关注列表

### `default`

- 平衡的修剪队列
- 平衡的保留列表
- 排名的添加/关注队列
- 在有用处起草温暖介绍或直接外联

### `aggressive`

- 更大的修剪队列
- 对过期的未回关容忍度更低
- 仍然在应用前经过审查关卡

## 评分模型

使用以下正面信号：

- 互惠性
- 近期活跃度
- 与当前优先级的一致性
- 网络桥接价值
- 角色相关性
- 真实的互动历史
- 近期存在感和响应性

使用以下负面信号：

- 已消失或废弃的账号
- 过期的单向关注
- 偏离优先级的主题集群
- 低价值噪声
- 反复未响应
- 存在很多更好替代时仍不回关

互关和真正的温暖路径桥接应比单向关注受惩罚更少。

## 工作流

1. 捕获优先级、不触碰约束和选择的平台。
2. 拉取当前的关注/连接清单。
3. 用明确原因对修剪候选者评分。
4. 用明确原因对保留候选者评分。
5. 使用 `lead-intelligence` 加研究表面排名扩展候选者。
6. 匹配正确的渠道：
   - X DM 用于温暖、快速的社交触点
   - LinkedIn 消息用于专业图谱邻近性
   - Apple Mail 草稿用于更高上下文的介绍或外联
7. 在起草消息之前运行 `brand-voice`。
8. 在任何应用步骤之前返回审查包。

## 外联规则

- 默认邮件路径是 Apple Mail / Mail.app 草稿创建。
- 不自动发送。
- 根据温暖度、相关性和上下文深度选择渠道。
- 当邮件或无外联是正确选择时，不强制使用 DM。
- 草稿应该听起来像用户本人，而非自动化的销售文案。

## 相关技能

- `brand-voice` 用于可复用的声音画像
- `social-graph-ranker` 用于独立的桥接评分和温暖路径数学
- `lead-intelligence` 用于加权目标和温暖路径发现
- `x-api` 用于 X 图谱访问、起草和可选的应用流程
- `content-engine` 当用户还想要围绕网络变动的公开发布内容时
